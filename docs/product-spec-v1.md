# Product Spec v1

## 1. Цель продукта
Единая платформа для управления детскими занятиями для:
- модератора платформы;
- администраторов секций;
- тренеров;
- родителей.

Приложение работает в двух клиентах одновременно:
- Telegram Mini App + бот;
- PWA (iOS/Android/desktop).

Источник истины один: backend + единая БД.

## 2. Роли и права

### 2.1 `super_admin`
- Управление секциями и пользователями секций.
- Назначение/снятие ролей `section_admin` и `coach`.
- Глобальные настройки платформы.
- Доступ к аудиту действий и журналу доставки уведомлений.

### 2.2 `section_admin`
- Управление секцией: группы, расписание, тренеры, дети.
- Настройка правил списания/продления абонементов.
- Настройка каналов оплаты секции.
- Отправка уведомлений по секции.

### 2.3 `coach`
- Доступ только к назначенным группам/занятиям.
- Подтверждение посещаемости.
- Подтверждение/отклонение отсутствий.
- Подтверждение/отклонение ручных оплат (`manual_transfer`).

### 2.4 `parent`
- Доступ только к своим детям.
- Просмотр расписания, абонементов, оплат.
- Подача отсутствия ребенка.
- Оплата доступными каналами секции.

## 3. Модель контекста родителя
- У родителя может быть несколько детей.
- Один ребенок может ходить в несколько секций и групп.
- В интерфейсе родителя используется контекст:
  - активный ребенок;
  - активная секция (или "Все секции").
- Все экраны родителя фильтруются по выбранному контексту.

## 4. Функциональные требования MVP

### 4.1 Посещаемость
- Посещаемость подтверждает тренер.
- Быстрый режим учета:
  - действия по ребенку: `present`, `absent`, `late` (опционально);
  - массовые действия: "отметить всех как `present`";
  - поиск по списку детей;
  - автосохранение без дополнительных модалок.

### 4.2 Статус ребенка на занятии для родителя
Родитель видит статус в реальном времени:
- `expected` (ожидается);
- `in_class` (на занятии);
- `absent` (отсутствует);
- `finished` (занятие завершено).

### 4.3 Отсутствия
- Инициатор отсутствия: родитель или тренер.
- Решение по легитимности принимает тренер (или админ секции).
- Статусы отсутствия:
  - `pending`;
  - `approved`;
  - `rejected`.

Правило списания:
- При `approved`: не списывать занятие и/или продлевать абонемент (по настройке секции).
- При `rejected`: списывать по стандартному правилу.

### 4.4 Абонементы
Типы:
- `by_sessions` (по количеству занятий);
- `by_time` (по времени, например минуты/часы).

Списание применяется после финального статуса посещаемости/отсутствия.

### 4.5 Оплаты
Каналы оплаты:
- `auto_link` (ссылка СБП);
- `auto_qr` (СБП QR);
- `manual_transfer` (ручной перевод).

Потоки:
- `auto_link`/`auto_qr`: подтверждение через webhook провайдера, затем автоматический учет.
- `manual_transfer`: родитель отправляет сумму и, при необходимости, скриншот; тренер подтверждает или отклоняет.

### 4.6 Уведомления
- Создает: `section_admin` или `coach` (в рамках доступа).
- Типы:
  - напоминание о занятии;
  - изменения в расписании;
  - игры/мероприятия;
  - подтверждение оплаты;
  - решение по отсутствию.
- Доставка: одновременно в Telegram и PWA.
- Ведение журнала доставки с ретраями.

## 6. Аутентификация и сессии
- Telegram: авторизация через Telegram Mini App init data.
- PWA: одноразовый вход (magic link/SMS OTP), далее refresh token.
- Пользователь не вводит логин/пароль на каждом входе.

## 6.1 Invite onboarding (Telegram + PWA)
- Единая invite-ссылка на секцию: `https://app.vneclassno.ru/invite/{token}`.
- Один и тот же `token` валиден для Telegram и PWA.
- Если ссылка открыта в Telegram-контексте, пользователь переходит в бот/mini app с тем же `token`.
- Если ссылка открыта вне Telegram, пользователь остается в PWA и проходит тот же onboarding.

Сценарии:
- Пользователь авторизован и ребенок уже создан: выбор ребенка и отправка заявки на добавление в секцию.
- Пользователь авторизован, но ребенка еще нет: создание карточки ребенка и отправка заявки.
- Пользователь не авторизован: сначала вход, затем возврат на тот же `/invite/{token}`.

Кто может создать invite-ссылку:
- `section_admin` и `coach`.
- `parent`, если это разрешено настройкой секции.

Принятие в секцию:
- `auto_accept` (автодобавление), либо
- `pending` с подтверждением `section_admin`/`coach`.

## 7. Нефункциональные требования MVP
- Полная синхронизация Telegram/PWA через единый backend.
- Событийная доставка уведомлений с ретраями.
- Аудит критичных действий (посещаемость, решения по отсутствию, подтверждение оплат).
- Время отклика для массовой отметки посещаемости: целевой p95 до 500 мс на запрос.

## 8. Критерии готовности MVP
- Все роли и доступы работают по политике прав.
- Родитель видит актуальный статус ребенка на занятии.
- Абонементы корректно списываются/не списываются по правилам отсутствия.
- Все три канала оплаты работают, включая ручное подтверждение.
- Уведомления доставляются в Telegram и PWA из одного действия.
